# 变异测试设计

> 本项目的工具已完成「大模型自动生成测试场景 + ReAct 智能体自动执行 + 独立判官判定 PASS/FAIL」的全链路。**变异测试是这条链路的评估层**：它不测应用，而是给「判官」这个测试预言（test oracle）打分。本文是变异测试的设计说明。

---

## 一、要解决什么

任务二用一个独立的 LLM 判官给每次执行下 PASS/FAIL。但判官本身准不准，**没有 ground truth 可以对**——这就是软件测试里经典的 **test oracle 问题**。

判官会不会把失败的判成通过？会不会死抠字面误杀真实的成功？靠人工抽查既慢又主观。我们需要一种**可量化、可复现、自动化**的方法评估判官质量。

变异测试（mutation testing）正是 oracle 评估的标准手段：**主动往「正确的执行」里注入已知的错误，看 oracle 能不能检出**。检出了说明 oracle 可靠，检不出就精确定位它的弱点。

> 与传统变异测试的区别：传统变异测试测的是**测试用例**的质量（好测试该杀掉被改坏的程序）；本设计测的是**判官**这个 oracle 的质量。被测对象从「测试套件」换成了「LLM 判官」。

---

## 二、设计目标

1. **打分**——给判官一个可量化的质量分数（Mutation Score）。
2. **定位**——指出判官在哪类错误上可靠、哪类上失灵（分维度报告）。
3. **驱动改进**——用分数和弱点清单反过来指导判官加固，形成闭环。
4. **低成本**——评估过程要能反复跑，不能每次都重开浏览器。

---

## 三、核心设计

### 3.1 被测对象是判官，不是应用

关键设定：**应用恒正确，被测 = 判官**。

正确的测试场景在 demo 站上真实跑出一条正确的执行轨迹（trace）。我们在这条 trace 上动手脚制造「错误」，再让判官去判。判官判 **FAIL = 检出（killed）**，判 **PASS = 漏检（survived）**。于是判官的质量 = 它杀掉多少个已知错误。

这避免了「应用本身可能有 bug」的干扰——我们不关心应用对不对，只关心判官对「判错」敏不敏感。

### 3.2 两层变异，分别回答两个问题

一个执行结果有两侧：**规约（场景的 expectation）**和**行为（实际 trace）**。判官可能只看一侧。于是设计两层互补的变异：

| | Layer 1：规约变异（spec） | Layer 2：行为变异（trace） |
|---|---|---|
| 改什么 | 改场景的 expectation | 改执行轨迹的步骤结果 / 页面观察 |
| 保持什么 | trace 不动（行为是对的） | 场景 expectation 不动（规约是对的） |
| 现实含义 | 「把目标写错了」——文档/需求被曲解 | 「应用表现错了」——真出了 bug |
| 回答的问题 | 判官**看不看规约**？ | 判官**看不看实际行为**？ |

两层都跑，才能区分两种漏检：一个连规约改了都无动于衷的判官，和一个只认规约不看行为的判官，弱点完全不同。

### 3.3 复用轨迹模型（成本关键）

判官本质是个纯函数：`verdict = judge(场景, 轨迹)`。隔离它**只需要换输入，不需要重跑浏览器**。

所以评估流程是：

```
正确场景 ──真跑 1 次──▶ 真实轨迹
                           │
              ┌────────────┴────────────┐
              ▼                         ▼
       生成 N 个变异体            （同一份轨迹）
              │                         │
              └────────────┬────────────┘
                           ▼
              判官对每个变异体重判（N 次纯 LLM 调用）
                           ▼
                      Mutation Score
```

成本从「变异体数 × 浏览器」降到「1 × 浏览器 + N × LLM」。一次真跑的轨迹，可以喂给几十上百个变异体反复判。这让「跑一次完整评估」从几小时缩到几分钟。

### 3.4 双档判官，量化宽松代价

判官的「严」是个连续谱：太松会漏检真错误，太严会误杀真成功。我们同时跑两档判官在同一批变异体上：

- **lenient（宽松）**：抓实质，不死抠字面，核心目标达成即 PASS。
- **strict（严格）**：逐条核对 expectation，终态优先，矛盾即 FAIL。

两档的对比本身就是结论：strict 比 lenient 多杀掉的变异体（strict-only kills）就是「宽松判官漏掉的真错误」；而 strict 在真实成功场景上比 lenient 多杀掉的部分，就是「严格的代价」（误杀）。这样「严不严」不再是感觉，而是一张 trade-off 表。

## 四、变异算子设计

### 4.1 设计原则

- **规则式、可复现、不调 LLM**：算子用代码生成变异体（字符串替换、证据改写），不靠 LLM。否则「用 LLM 评 LLM」会引入新的不可靠。
- **互斥分区**：词表按算子切分，同一个词不会被多个算子重复改，保证变异体不重叠。
- **must-kill vs soft 分类**：与真实轨迹**正面矛盾**的变异体（实体错、数量错、存在性取反）是 must-kill——判官理应必杀；纯描述性漂移（措辞、视觉状态）是 soft——宽松判官允许存活。分桶报分，避免 soft 拉低分数造成误判。
- **故障必须自洽**：Layer 2 的故障不能只改一处、留下大量矛盾的成功证据——宽松判官会把矛盾往「成功」方向和解。改一处要连同相关证据一并改干净。

### 4.2 Layer 1：规约变异算子

对每个含 expectation 的段落，作用 6 类算子，只改 expectation、不动步骤：

| 算子 | 类别 | 作用 |
|---|---|---|
| negate | must-kill | 取反/反义：出现↔消失、添加↔删除、启用↔禁用 |
| entity-swap | must-kill | 实体循环：看板→卡片→列表→看板；模板 Simple↔Kanban |
| number-tamper | must-kill | 数值篡改：阿拉伯/中文数字改成不同值 |
| state-swap | must-kill | 状态取反：升序↔降序、勾选↔未勾选、暗色↔亮色 |
| keyword-drift | soft | 近义漂移：卡片→任务、看板→工作区 |
| feature-add-del | soft | 增删一条 key_feature（如插入 trace 里不会有的 toast） |

### 4.3 Layer 2：行为故障注入

在真实 trace 上注入 3 类**自洽**故障，场景 expectation 保持正确：

| 故障 | 含义 | 自洽手法 |
|---|---|---|
| exec-failure | 执行异常：创建步骤失败 + 抹掉资源证据 | 步骤结果改失败，并从所有页面观察里删掉该资源，一致呈现「未创建」 |
| layout-missing | 布局缺失：动作成功但未渲染 | 仅从最终观察抹掉资源，步骤结果保持成功 |
| semantic-flip | 语义翻转：终态状态被翻转 | 最终观察里的状态/反义词翻转（升序↔降序、出现↔消失） |

> 三类故障都用本场景创建的资源名（带命名空间前缀）来精确定位「该动哪段证据」，避免无差别破坏。

## 五、评估指标

```
Mutation Score = killed / total                         （total=0 记 0）
must-kill Score = killed_must / total_must              （核心指标）
spec-sensitivity ≈ Layer 1 的 must-kill Score           （判官对规约的敏感度）
strict-only kills = lenient 漏检而 strict 抓到的数量      （宽松判官的盲区锚点）
宽松代价 = strict 在真实成功场景上误杀的数量              （严格的代价）
```

- **must-kill Score** 是主指标：判官对这些「理应必杀」的错误检出多少，直接反映它可不可靠。
- **soft Score** 是参考：宽松判官 soft 存活是设计内的，不算缺陷。
- **spec-sensitivity** 专门刻画「判官到底看不看 expectation」——这是本项目的核心发现之一。
- **宽松代价**用同一批真实成功场景，分别让两档判官判，比他们的 PASS 率差。

---

## 六、执行流程

**Layer 1（规约变异）**：
```
原场景 ──真跑──▶ 真实轨迹 trace
                    │
       生成 N 个变异体（每个改了某段 expectation）
                    │
       对每个变异体：judge(变异场景, 同一份 trace)
                    │
       verdict = FAIL → killed；PASS → survived
                    │
              汇总 must-kill Score（= spec-sensitivity）
```

**Layer 2（行为故障）**：
```
原场景 ──真跑──▶ 真实轨迹 trace
                    │
       注入 M 个自洽故障（每个改了 trace 的证据）
                    │
       对每个故障轨迹：judge(原场景, 故障轨迹)
                    │
       verdict = FAIL → killed；PASS → survived
                    │
              汇总 must-kill Score
```

两层数据都来自**同一次真跑**，复用轨迹模型让评估成本可控。

## 七、实测结果

> 以下为现有实现跑出的真实数据。

| 维度 | 结果 | 说明 |
|---|---|---|
| Layer 1 spec-sensitivity（lenient） | **≈ 0%** | 宽松判官几乎不看 expectation，靠标题+步骤+轨迹推断目标，goal 级全改也存活 |
| Layer 1 must-kill（strict） | lenient 0% → **strict 38%** | 严格判官对实体/状态/数量类错误有敏感度，但 negate 类仍 0% |
| Layer 2 Mutation Score | **33%** (3/9) | 整体检出率 |
| Layer 2 分化 | 薄证据/单动作 **100%**；富证据/多步 **0%** | 富证据场景里，单点故障被其他幸存证据「救回」 |
| 宽松代价 | 真实 PASS 率 lenient 46% / strict 25% | strict 误杀约 7 个真实通过（−21pp） |

**核心结论**：判官「**不查规约、查行为；行为故障的检出依赖证据是否冗余**」。证据薄的单动作场景（排序、主题、开关），判官一抓一个准；证据丰富的多步场景（建看板/卡片），任何一个证据通道幸存，判官就倾向判 PASS。

---

## 八、局限与诚实标注

- **富证据和解**：Layer 2 对富证据场景检出率为 0，因为判官做多源证据「和解」，要所有证据通道同时失败才 FAIL。这是判官的设计取舍，也是当前评估暴露的最主要弱点。
- **negate 类盲区**：连严格判官对「不应出现 X」类期望也检不出（0%）。纯 LLM 对否定表达不可靠。
- **故障适用面**：layout-missing 只对创建类场景有效（需要资源出现在最终观察里）；semantic-flip 在最终观察多为英文可访问性树时，中文状态词命中率低。
- **被测边界**：评估的是判官，不评估应用本身、也不评估 ReAct 执行器。判官与执行器共享同一个 LLM（DeepSeek），存在「同源偏差」的潜在风险——未来可换用异构模型作判官交叉验证。

这些局限不是缺陷报告，而是评估本身交付的一部分：**变异测试的价值就是把「判官准不准」从主观印象变成可定位的、可答辩的事实**。
