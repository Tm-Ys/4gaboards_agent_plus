## 二、阶段性记录

### 任务一：测试场景自动生成

- **双通道知识源**：用户手册 `4gaBoardsDocs/docs/*.md`（account / project / board / list / card / view / sidebar / notifications / settings / import-export / shortcuts …）+ 源码 `4gaBoards/`（理解真实 REST 接口与数据模型）。
- **检索策略定为「长上下文直填」而非 RAG**：RAG 会检索到语义相近但不相同的内容引入噪声；1M 上下文够用，直接把对应文档整段送入。
- **生成流程**：① 功能点提取（按模块逐篇提取后合并去重，**合并同族微操作**——富文本工具栏按能力归并、快捷键集合、同质设置开关同理）；② 场景生成（每功能点 + 来源文档全文 → 1~3 个场景，必含 happy_path，文档支撑时补 variant/edge_case）。
- **成果**：**118 功能点 / 179 场景 / 100% 覆盖**，固化为默认集 `basic`（入库），任务二无需每次重新生成。
- **双语言**：Python 原型 `scenario_generator/`（已验证）+ TS 主交付 `app/`（prompt / schema 完全一致、产物互通）。

### P0–P3：ReAct Agent + 判官 + 批量 harness

- **P0**：Playwright + 观察构建器（AX→ref 文本，含 heading）+ 工具注册框架 + B 层通用工具（`click / fill / press / scroll / goto / observe / done`）+ A 层种子（`auth_login / board_create / board_open / card_create`）。
- **P1**：ReAct 循环（function-calling ↔ registry）跑通 board-create happy_path；agent 能自适应真实 UI、失败自恢复。
- **P2**：独立 LLM 判官（场景级终判 PASS/FAIL + 失败步骤 + 原因）。trace 机制让多步封装工具对判官可见；判官抓实质——核心达成即 PASS，描述性/语言不匹配项记 missed 不翻盘。
- **P3**：批量 harness（`run-batch`）+ 通过率报告。**基线 43%**（30 个 easy+happy_path：board 5/5、admin 3/3、account 1/1 全过；**instance 设置开关类 0/13 全挂**）。

### P1.5：preferences 设置簇（settings 68%）

- 侦察证伪了原假设——**instance 开关簇被 demo 站 demoMode 物理禁用、不可救**（已文档化为环境受限）；转向 preferences 簇，新增 `settings_open` / `settings_toggle` / `settings_select` 工具，settings 簇批量通过率 **68%**。

### P4：健壮性（settings 68% → 84%）

1. **状态隔离** `resetAccountLanguage`：每场景恢复账号级 state（如语言），settings 簇 **68%→84%**。
2. **命名空间 + REST 清理** `cleanupTestProjects`：按 `${ns}-` 前缀删测试 board/project。
3. **拖拽** `card_drag`：react-beautiful-dnd 的 mouse 多步 move（看板核心交互，非 click / dragAndDown）。

### card 工具补全（A 层 18 → 20）

补齐 manage_labels / text_view 两个簇：

- **`card_manage_labels`**（toggle / create / edit）：4gaBoards 标签是**彩色按钮**（选中态 = `nameActive` 类，非 checkbox）；弹层搜索框占位符实测 `"Search labels or create one..."`；标签项定位用 `button[title="<name>"]`（title=标签名，比 role name 稳）；toggle 选中态经 **socket 往返**更新，需**轮询 ~4s** 读 class 变化（弱网下 700ms 不够）；create 后标签自动选中。
- **`card_text_editor`**（switch_mode / resize / help）：@uiw/react-md-editor@4。模式由根节点 class 指示 `w-md-editor-show-edit/live/preview`；Ctrl+7/8/9/0 在本版本未绑定 → 改点**工具栏右侧模式按钮**；resize 手柄 = `.w-md-editor-bar`；help 按钮 title="Open help"。
- **helper**：抽 `enterDescriptionEdit`、`readEditorMode`。
- 五场景验证：text-view-switch-modes ✅、text-view-resize ✅、manage-labels-2 ✅、**open-help ❌**（@uiw `commands.help` 的 window.open 在 headless 不产生可追踪 popup，环境受限）、**manage-labels-1 ❌**（复杂 medium 多步编排）。

### P5 变异测试（2026-06-18）—— 给独立判官打分

被测 = 判官，app 恒正确。代码 `app/src/agent/mutation/`，CLI `run-mutation --layer spec|trace`。

- **Layer1（改 expectation，真跑正确 app，重判官）= spec-sensitivity ≈ 0%**——判官无视 expectation、靠 title+steps+trace 推断核心目标。把 title+desc+expectation+steps 整体改成另一目标（goal-level）也 0/3 全存活，且会**主动推翻亲眼看到的 spec 矛盾**（Simple vs Kanban：判官自述「实际选了 Simple 而非指定的 Kanban，但核心目标达成」）。
- **Layer2（往真实轨迹注入故障、用原场景重判官）= 3/9 (33%)**——判官做**多源证据和解**，单个失败步骤会被幸存证据救回（card-create 自述「card_create 首次失败，但列表卡片数 2→3，创建成功」）。分化：薄证据/单动作场景（list-sort / settings-theme / sidebar-toggle）**100% killed**；富证据/多步（board / card / view / notifications）**0%**。
- **结论**：判官「**不查规约、查行为；行为故障检出依赖证据是否冗余**」——富证据场景要所有证据通道同时失败才会 FAIL。

### P6 判官加固（2026-06-18）—— 量化宽松代价

- **strict 判官变体**（`judge.ts` 加 `JudgeMode`：逐条核对 expectation + 终态优先历史不救 + must-have/nice-to-have 分类 + 状态可表达性降级 + 跨语言强映射），与既有 lenient 并存，默认 lenient 不破坏既有行为。
- **`--judge both`** 两判官对比：runner 加 `baselineOverride` 复用基线 trace（浏览器只跑一次、判官跑两遍）；`report.ts` 加 `compareJudges`（strict-only kills = 宽松漏检锚点）。
- **实测（Layer1 board-create）**：lenient **0/14（must-kill 0%）** → strict **8/14（must-kill 38%）**。按算子 entity-swap 0→67%、state-swap 0→100%、soft 类 0→75-100%；**negate 仍 0%（strict 弱点，待 prompt 调优）**。
- **宽松代价（30 场景零浏览器重判）**：真实 PASS 率 lenient **46%** / strict **25%**，**strict 误杀 7 个真实通过（-21pp）**。误杀里 ~2 个是真问题（board-view-toggle 终态视图、board-export-csv 操作路径偏离）、~5 个 strict 太严（headless 下载无 UI 提示 / 终态本就该关 / demo 无数据）。
- **健壮性修复**：DeepSeek 偶发畸形 JSON 曾让 `--judge both`（调用翻倍）整批崩 → `chatJson` 加 3 次重试 + 变异体/故障循环单条容错（跳过失败项不崩、不丢已跑结果）。
- **结论**：lenient 高通过率但零规约敏感；strict 有规约敏感但误杀真实通过——这张 trade-off 三角就是「宽松代价」的量化交付。
- 代码：`verify/judge.ts` + `mutation/{runMutation,runMutationTrace,report}` + `cli/{run-mutation,run-judge-cost}` + `llm.ts`。

### 变异测试复现验证（2026-07-07）

对照《[变异测试设计](变异测试设计.md)》文档实跑 `board-create-happy-path × --judge both`，确认 P5/P6 结论稳定可复现，全链路（算子 + 复用轨迹 + 双判官 + 对比报告）可用：

- **Layer1（改 expectation）**：14 个变异体，lenient must-kill **0/8 (0%)** → strict **3/8 (38%)**，与 P6 的 0%→38% 完全一致。按算子：entity-swap 0→67%、feature-add-del 0→50%、negate 0→25%（取反类仍是 strict 弱点，与既有结论吻合）。lenient 对全部 14 个变异体零检出，干净印证「判官不查规约、查行为」。
- **Layer2（注入故障）**：board-create 单场景生成 1 个故障（exec-failure）；layout-missing / semantic-flip 因终态观察无 namespace 资源 / 无中文状态词而跳过——即设计文档 §8 标注的故障适用面局限。该 exec-failure 被 strict 检出、lenient 漏检，印证「lenient 富证据和解、strict 终态优先」。
- 报告落 `app/outputs/mutation/mutation-*.json`（gitignored，运行产物）。结论：变异测试设计落地、结论可复现。

## 三、已知问题（环境受限，非工具缺陷）

- **弱网**：批量 `page.goto` 登录/清理偶发 60s 超时；cleanup 语言恢复在弱网下脆弱。
- **open-help**：@uiw `commands.help` 的 `window.open` 在 headless 不产生 Playwright 可追踪 popup，环境受限（类比 instance demoMode）。
- **manage-labels-1**：复杂 medium 多步编排，create/toggle 已可用但全序列 + rename 对 agent 偏难。
- **P5 Layer2 的 semantic-flip**：finalObs 多英文/符号 AX 树，中文状态 token 难命中而偶发不触发；layout-missing 仅创建类场景适用（需 namespace 资源出现在 finalObs）。
- **判官严格性**：card-delete-via-bin（按 Delete 键 vs 点 bin 图标）、open-card-by-clicking-row（抓「点行空白区」操作方式）—— 核心目标其实达成。
- **card 簇剩余 FAIL**：open-help（环境受限）、manage-labels-1（复杂 medium）、view-activity-comment（波形图标）、card-edit-title / copy-link / check-activity（agent 未调用已有工具，编排问题非工具缺陷）。
